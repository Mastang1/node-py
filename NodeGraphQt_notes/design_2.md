### 2. 核心设计原则
- 一个 `NODE` 只表示 `@Instruments_pythonic` 中一个 `class` 的一个 `method`，禁止一个节点混入多个 method 语义。
- 仪器句柄必须设计为可流转的数据对象，可作为多个后续 API 的输入参数，也可作为 API 返回值 `list` 中的一个元素。
- API 节点显示的参数名、返回值名、中文标题、中文提示信息，统一来自每个 `method.__doc__` 中的 JSON 元数据，不允许在 UI 层手写散落映射。
- 鼠标悬停展示的功能描述，统一读取上述 JSON 中的 `func_desc` 字段。
- 运行时执行逻辑、保存/加载 JSON 逻辑、导出 Python 逻辑，三者必须共享同一份节点元数据模型，避免“界面能跑但导出代码不一致”的问题。
- 所有新增功能必须以“用户可理解、可恢复、可验证、可扩展”为准则，不做仅能演示但无法闭环的临时代码。

### 3. 建议的元数据规范
- 在每个 `method.__doc__` 中使用 JSON 字符串，至少包含以下字段：
- `node_title_zh`：节点中文标题
- `func_desc`：功能描述，用于 hover 提示
- `params`：参数列表，每项包含英文参数名、中文显示名、类型、是否必填、默认值、说明
- `returns`：返回值列表，每项包含英文名、中文显示名、类型、说明
- `category`：节点分类，用于节点列表展示与排序
- `icon`：对应的 SVG 图标标识
- 如果现有 `__doc__` 不满足该结构，则先做兼容解析层，再逐步推动标准化。

### 15. API 转 Node 原理说明（费曼学习法）

#### 15.1 先把它想成“说明书自动变积木”
- `Instruments_pythonic` 里的每个 `class.method`，都像是一张带说明书的小卡片。
- 这张卡片的说明书不是写给人看的长段文字，而是写在 `method.__doc__` 里的 JSON。
- 这个 JSON 里描述了：
- 这个节点叫什么中文名
- 它属于哪一类
- 它有哪些参数
- 每个参数的中文显示名、类型、默认值、提示信息
- 它会返回什么
- 返回值的中文显示名和类型
- 如果它是控制节点，还会告诉系统它有哪些控制流出口
- 程序做的事情，本质上就是：先读说明书，再按说明书把图形节点搭出来。

#### 15.2 为什么不能手写一堆节点
- 如果每个 API 都手工写一个 Node 类，会有几个问题：
- API 一多，维护量爆炸
- 一旦 API 参数改名，UI、执行器、导出器都要跟着手改
- 中文显示、tooltip、返回值端口很容易和真实 API 脱节
- 所以更合理的方式是：把 API 的“描述信息”收敛到 `__doc__` JSON 里，代码只负责解释 JSON。

#### 15.3 真实实现时，程序做了哪几步
1. 扫描 `demo_02/Instruments_pythonic` 下面的模块。
2. 找到模块里定义的类。
3. 找到类里的公开方法。
4. 读取每个方法的 `__doc__`。
5. 如果 `__doc__` 能解析成合法 JSON，就把它变成内部元数据对象。
6. 根据这份元数据动态创建 Node 类。
7. Node 类在初始化时，不再只创建一个普通流程端口，而是按元数据创建：
8. 流程输入端口
9. 流程输出端口
10. 中文数据输入端口
11. 中文数据输出端口
12. 同时保留属性面板里的参数作为“未连线时的默认值输入”。

#### 15.4 为什么现在节点上既有端口又保留属性
- 因为这两个东西解决的是两个不同问题：
- 数据端口：解决“上一个节点的结果如何流到下一个节点”
- 属性面板：解决“这个输入如果没有连线，用什么默认值”
- 你可以把它理解成：
- 连线输入是“外接数据源”
- 属性面板是“本地兜底值”
- 运行时优先读连线；如果这个参数没有连线，再回退到属性值。

#### 15.5 中文参数/返回值是怎么显示到节点上的
- JSON 里每个参数有 `name` 和 `zh`。
- `name` 是程序内部键，例如 `handle`、`frequency`。
- `zh` 是用户看到的中文端口名，例如“仪器句柄”“频率(Hz)”。
- 动态节点创建时：
- 内部逻辑按 `name` 工作
- 节点端口按 `zh` 展示
- 这样既保证程序稳定，也保证界面是中文可读的。

#### 15.6 仪器句柄为什么能真正流转
- 初始化类 API 例如 `initialize()` 会返回一个 `list`，其中第一个元素可能就是句柄。
- 系统会把返回值拆开，按 `returns` 里的定义，分别挂到不同的数据输出端口上。
- 如果某个返回值类型是 `handle`，那它就不只是一个普通字符串，而是被当成“后续 API 可以消费的句柄值”。
- 下一个节点如果某个参数类型也是 `handle`，它就可以直接从这个输出端口连过来。
- 所以这里不是“把 list 原样传来传去”，而是“把 list 按说明书拆成多个可连的数据口”。

#### 15.7 控制节点为什么也能从 `__doc__` 生成
- 普通仪器 API 只有一个默认的流程出口：执行完就往后走。
- 但控制节点不一样，比如 `if` 有两个出口：
- 条件成立
- 条件不成立
- 所以在 `general.py` 的控制类方法 `__doc__` JSON 里，又额外加了控制流元数据，例如：
- `control_kind`
- `flow_outputs`
- 这样系统看到它不是普通 API，而是“带分支语义的 API Node”。
- 于是动态 Node 工厂就会给它生成多个流程出口，而不是一个普通 `flow_out`。

#### 15.8 一句话总结
- API 转 Node 这件事，本质不是“写了很多节点类”，而是“写了一个会读说明书的节点工厂”。
- `__doc__` JSON 是设计图。
- 动态 Node 工厂是生产线。
- 生成出来的节点，就是图形化编程里真正能拖、能连、能执行的积木。

### 16. Python Node Flow 导出原理说明（费曼学习法）

#### 16.1 先把它想成“把画布翻译成脚本”
- 画布上你看到的是节点和连线。
- 但 Python 解释器看不懂“节点长什么样”，它只看得懂代码。
- 所以导出的本质工作就是：
- 把“节点图”翻译成“Python 脚本”
- 并且要保证这个脚本跑出来的行为，和 GUI 里点击运行时一致

#### 16.2 以前为什么顺序导出不够
- 当流程只是单链路时，可以很简单：
- 从前往后一个节点一个节点写代码
- 这就像把火车一节一节排成一条线
- 但一旦出现：
- `if / elif / else`
- `while`
- `for`
- 回边
- 多个流程出口
- 这种“只按线性顺序写代码”的方案就不够了，因为图已经不是一条直线，而是一个带分支和回路的状态图。

#### 16.3 这次为什么改成“状态机式导出”
- 现在的导出思路，不再假设图一定是直线。
- 它把每个节点导出成一个小函数。
- 每个小函数负责三件事：
1. 读取自己的输入
2. 执行自己的逻辑
3. 返回“下一步应该从哪个流程出口走”
- 外面再有一个总调度器，用 `while current_node_id:` 持续运行。
- 你可以把它想成：
- 每个节点像一个路口工作人员
- 它干完自己的事以后，会告诉调度器“下一步该走哪条路”
- 调度器再根据这条路跳到下一个节点函数

#### 16.4 为什么这种方式能支持分支和循环
- 因为状态机不要求图必须是直的。
- 它允许：
- 这个节点走到 A
- A 再走到 B
- B 再回到前面的节点
- 也允许：
- 同一个节点根据条件不同，走不同出口
- 所以它天然适合 Node Graph。
- GUI 运行时和导出脚本都遵循同一条规则：
- 节点执行完，不是“自动去下一个数组元素”
- 而是“根据当前节点返回的流程出口，去找下一个节点”

#### 16.5 数据端口在导出脚本里是怎么工作的
- 每个节点除了流程出口，还有数据输入输出。
- 导出脚本里专门维护了一个 `context['port_values']`。
- 你可以把它理解成“所有节点输出结果的寄存器表”。
- 当某个节点执行完，它会把自己的输出值写进去，例如：
- `(node_id, output_key) -> value`
- 当下游节点执行时：
- 如果这个参数有连线，就去 `port_values` 里拿上游结果
- 如果没有连线，就退回到节点属性里的默认值
- 所以导出脚本并不是“把值硬编码死”，而是在运行时模拟了一遍节点连线的数据传输。

#### 16.6 GUI 运行和导出脚本为什么能保持一致
- 因为两边现在都遵循同一套核心语义：
- 输入值优先取连线，再取属性默认值
- 控制流由节点返回的 `next_flow_key` 决定
- 数据输出按返回值列表拆分到对应输出端口
- 循环节点靠自身状态和回边再次进入
- 终止节点设置终止状态并结束后续执行
- 换句话说：
- GUI 里不是一套“看起来像流程图”的逻辑
- 导出脚本里也不是另一套“随便拼的 Python”
- 两者都在执行同一个抽象模型

#### 16.7 `for` 和 `while` 导出为什么没有崩
- 因为导出脚本不是简单把所有节点按顺序抄下来。
- 它会导出：
- 节点函数
- 流程出口映射表
- 当前节点 ID
- 循环步数保护
- 这样当 `for` 或 `while` 节点返回“去循环体”时，调度器就跳到循环体；
- 当循环体执行完又回到循环节点时，调度器继续跑；
- 当循环节点返回“完成”时，调度器再跳到完成出口。
- 这就是为什么循环图现在既能在 GUI 里跑，也能在导出的 Python 里跑。

#### 16.8 一句话总结
- Python node flow 导出，本质不是“把节点名字打印成几行代码”。
- 它真正做的是：
- 把节点图翻译成一个可执行状态机
- 把数据连线翻译成运行时数据表
- 把流程分支翻译成下一跳决策
- 所以导出脚本才能真正复现 GUI 里的节点行为，而不是只做个样子。

- `run_finished` / `run_failed` 与主窗口槽必须使用 **QueuedConnection**，避免在工作线程执行会触碰 GUI 的槽。

#### 17.4 通用「Python 代码」节点
- `PythonSnippetNode`（`demo_02/nodes.py`）：多行 `exec` 片段，`v1`/`v2` 输入与 `result` 输出；支持导出与校验（空代码报错）。交互类似 LabVIEW 公式节点，但语义为完整 Python 块，需注意安全与隔离策略（生产环境应配合白名单/沙箱）。

#### 17.5 Headless 冒烟测试与管道死锁
- `python demo_02.py --headless-smoke-test` 会设置 `DEMO02_QUIET_SMOKE=1`：关闭 `log()` 的终端打印与部分 UI 日志写入、仪器模拟器的 `iv_print`、以及 headless 子进程中的逐条 JSON 日志，避免 stdout 管道被灌满后 **子进程写阻塞 + 父进程读阻塞** 的死锁。
- 验证导出脚本时用 `subprocess.run(..., capture_output=True)` 承接子进程全部输出，避免同样问题。
- 在 QUIET 模式下 **跳过** `_on_debug_paused` 中的语法高亮与画布选中（Pygments + 图编辑在 offscreen + `processEvents` 轮询下易卡死）；调试语义仍在线程内执行，仅 headless 冒烟不依赖该段 UI。

### 20. 校验顺序与导出、资源树（2026-03 修正）

#### 20.1 流程校验 / 导出「新流程不可用」根因
- `analyze_flow_graph` 曾将 `ordered_nodes` 设为**按画布坐标排序的可达节点**，而会话绑定校验按该列表顺序模拟 open → use → close。若用户在画布上把「关闭会话」放在「打开」上方，会得到**误报**（如「引用了未打开的会话」），从而阻断校验、导出与预览。
- **修复**：`ordered_nodes` 改为从 **Start** 出发的 **DFS 栈遍历发现顺序**（与运行时探索顺序一致）；`reachable_nodes` 仍为稳定坐标排序，供导出与日志列举。

#### 20.2 导出源码拼接
- `WorkflowExporter.render_code` 不再丢弃列表中的空字符串行，避免意外合并语句行（保持生成脚本结构与可读性）。

#### 20.2.1 导出形态：仅函数体（测试集成）
- 导出结果为单个 **`instrument_flow_body(context)`**（可改名嵌入），内含嵌套节点函数与调度循环；**不再**生成独立可执行脚本（无顶层 `sys.path`/`main`）。
- 建议 import 以注释块 `__DEMO02_EXPORT_IMPORTS__` 标注，供复制到宿主模块。
- 详细原理见仓库 **`development_doc.md`**。

#### 20.3 仪器 API 与左侧资源树
- 动态仪器节点元数据来自 `discover_api_method_metas()` 扫描 `demo_02.Instruments_pythonic`（含 `generated/api_discovery_cache.json` 缓存）。**启动完成后**即调用 `ensure_instrument_api_registered` 并 `resource_tree.rebuild()`，左侧会包含扫描到的 API 模板；非写死列表。静态节点（开始/变量/注释等）仍来自 `nodes.py` 注册。
